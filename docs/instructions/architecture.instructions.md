# Architecture boundaries

- Main thread owns UI only. Do not run model inference, FFT-heavy work, large hashing, or long synchronous loops on it.
- `AudioWorkletProcessor` owns capture and enqueue only. It must not fetch, decode tokens, allocate large buffers repeatedly, or invoke ONNX Runtime.
- Dedicated ASR worker owns model sessions, decoder state, feature state, vocabulary automata, active profile state, and timing instrumentation.
- Enrollment/profile workers own recording analysis, sentence coverage, alignment, embedding aggregation, profile packaging, and future adapter preparation.
- Browser training must run behind the repository-owned `@speech/browser-training` `BrowserTrainingBackend` from ADR 0003 in its own dedicated training worker and never share the UI thread, AudioWorklet, or real-time ASR worker. UI surfaces may start/observe the worker, render only the user-facing Preparing/Training/Checking/Ready stages by default, request pause/cancel/restart/resume, and resume from a validated checkpoint, but must receive only aggregate progress/results for private datasets; technical epoch/loss/backend/thread/memory/checkpoint data belongs in explicit Training details. Cross-tab coordination belongs at the worker/client boundary using Web Locks plus BroadcastChannel so only one tab trains a given redacted profile/dataset scope at a time. ASR runtime activity has priority: training workers must subscribe to redacted ASR-priority events, pause cooperatively at safe boundaries, checkpoint, and release training resources rather than competing with live transcription.
- Service worker owns app-shell caching only; model/profile lifecycle belongs to dedicated stores and workers.
- Model-pack, worker-protocol, vocabulary, enrollment, and profile contracts must be UI-independent and versioned.
- v0.6 task-first UI work should use the committed v0.5 baseline inventory in `docs/planning/v0.6.0-ui-inventory.json` plus ADR 0008 as its parity checklist before moving, hiding, or deleting routes, actions, workflow states, or user-facing strings. ADR 0010 owns the v0.6 information architecture: exactly three persistent primary destinations (`Dictate`, `Vocabulary`, `Models`), app-menu ownership for settings/storage/privacy/shortcuts/diagnostics/about, no dashboard/home-screen detour, and no domain/worker contract changes for route work. ADR 0011 owns progressive-disclosure and minimal-copy boundaries: required actions, blockers, destructive/privacy consequences, and recovery stay visible at the point of need; optional diagnostics/expert detail may move into accessible disclosures, menus, dialogs, or dedicated screens. ADR 0012 plus `docs/planning/v0.6.0-terminology-copy-budgets.json` own v0.6 default UI terminology and copy budgets; default workflows should use mapped terms such as Voice model, Speech model, Vocabulary, Quality check, General speech check, Loud, Processing mode, and Device storage unless an explicitly advanced/diagnostic/legal/provenance context requires technical language. ADR 0013 plus `docs/planning/v0.6.0-route-map.json` own target routes, legacy v0.5 hash-anchor redirects, safe query preservation, focus/scroll expectations, and return-target validation before router implementation. `@speech/ui` owns semantic v0.6 visual tokens plus repository-owned Button/IconButton, Disclosure, Accordion, MenuButton, Tooltip, Dialog, Select, and RadioGroup primitives; app CSS may alias legacy variables to those tokens, but new primitives should consume `--speech-*` spacing, typography, colour, motion, elevation, focus, forced-colours, touch-target, and responsive tokens directly. `@speech/ui/testing` owns deterministic primitive accessibility examples, keyboard/focus matrices, and static audit helpers so feature-screen tests can reuse the primitive contract without importing domain workers, storage, audio, archive, encryption, or profile fixtures. Button/IconButton primitives should render native `<button>` elements, keep visible labels for ordinary actions, require `aria-label` for icon-only actions, and avoid manual ARIA state in application code. Disclosure should use native `<details>/<summary>` for one optional section; Accordion should use labelled heading buttons with hidden collapsed panels for several independent optional sections. MenuButton should render a native trigger with `aria-haspopup="menu"`, role-based action/navigation items, no nested submenu support, and keyboard/Escape/focus-return helpers. Tooltip should connect plain supplemental text with `aria-describedby`, open on focus/hover, dismiss on Escape, and never own required instructions or consequences. Dialog should be reserved for short blocking decisions, concise rename/passphrase interactions, and confirmations that fit one context; long workflows belong on dedicated routes. Select and RadioGroup primitives should keep persistent visible labels, native form semantics, validation descriptions, and keyboard support. Feedback primitives (`InlineError`, `Notice`, `Toast`, `EmptyState`, `LoadingState`, `Progress`, `Status`) should keep blockers, field recovery, progress labels, and state text visible with restrained live regions; toasts are transient confirmations only, while page blockers and destructive/privacy/import/training failures require visible notices or route content. Do not nest accordion patterns or put required fields, blockers, privacy/destructive consequences, or recovery actions only inside collapsed panels, menus, tooltips, toasts, or advanced dialog content.
- `apps/web/src/state/uiPreferences.ts` owns the additive v0.6 `UiPreferencesV1` record. Keep it UI-only, independently namespaced at `speech:ui-preferences:v1`, rollback-safe for v0.5 shells, and separate from profile/model/vocabulary/training stores; route/shell code may consume it, but domain packages and portable exports must not depend on it.
- `apps/web/src/app/AppShell.tsx` and `routeState.ts` own the initial v0.6 shell/navigation adapter. Keep existing panels mounted while the route layer migrates from v0.5 hash anchors to v0.6 path routes so workflow state is preserved, except dedicated settings/app-menu routes may render focused screens instead of the primary workflow stack. Route migration and navigation guards must go through typed `routeState` helpers: resolve target routes from the ADR 0013 contract, migrate legacy hashes, sanitize safe query/return-target state, expose focus/scroll restoration plans, and produce confirmation decisions for disruptive recording/enrollment/training/import/export/model-update navigation without changing domain contracts. Desktop and narrow navigation must keep visible labels, skip-link access, safe-area padding, heading focus restoration, and no domain worker/storage coupling. The shell app menu owns low-frequency Settings, Storage, Privacy, Shortcuts, Diagnostics, About, and update actions; Local status may observe PWA/model lifecycle through aggregate worker summaries only and must not become a diagnostics/storage panel. Settings index screens should be simple route lists, not one dense accordion page; `/settings/audio` reuses the audio capture controller only for an explicit input test and keeps diagnostics disclosed. `/settings/storage` owns aggregate storage summaries and deletion confirmations only; model/profile/training deletion remains worker/package-owned through existing lifecycle boundaries. `/settings/privacy` and `/settings/shortcuts` are static, dedicated app-menu screens: they may link to existing export/delete/diagnostics/docs routes, but must not import domain workers or become diagnostics panels. `/settings/diagnostics` owns dense grouped diagnostics and benchmark/support-bundle downloads; `/about` owns version, source, license, model provenance, acknowledgements, and update state without marketing copy. The Dictate workspace should keep capture/worklet/inference contracts untouched while presenting only the task toolbar, editable transcript, one recording control, state-relevant transcript actions, and a single technical-details disclosure by default. If the required base speech model is missing, Dictate owns only the setup presentation; catalog inspection, manifest sizing, install, verification, activation, retry, and cleanup progress stay in the existing model-lifecycle worker/model-manager boundary.
- Personal Models UI may render navigation, compact voice-model rows, aggregate capability/readiness preflight, profile/vocabulary/base-model status, accessible training controls/live regions/text equivalents, and user controls, but profile import/export/delete/activation/deactivation/rollback/duplicate, model manifest inspection, runtime self-tests, and heavy profile reads remain worker/package-owned. Model-list rows should show display name, active/generic state, actionable status, one state-dependent primary action, and row-menu low-frequency actions without raw IDs, hashes, storage paths, or technical metrics.
- Keep WebGPU, shared memory, graph capture, and FP16/INT8 as optimizations; correctness must preserve the WASM and transferable-buffer paths.

- The v0.6 component gallery is a development-only `#ui-gallery` route accepted by ADR 0014 and `docs/planning/v0.6.0-component-gallery-contract.json`. Keep it lazy-loaded behind `import.meta.env.DEV`, unlinked from production navigation, and backed by synthetic `@speech/ui/testing` examples only; do not import domain workers, storage, audio, archive, encryption, profile, transcript, vocabulary, or model fixtures into the gallery.

- #250 owns `docs/planning/v0.6.0-copy-deletion-pass.json`: default UI/status copy should stay task-first, and storage/runtime/checksum/provider/profile identifiers should appear only in explicit detail/diagnostic contexts.
- #251 owns `docs/planning/v0.6.0-content-budget-accessible-name-gates.json` and `apps/web/e2e/content-budget-accessibility.spec.ts`: critical task surfaces must remain semantically budgeted, primary actions and visible controls must have accessible names, and default workflows must avoid unexplained internal terminology without relying on screenshots as assertions.
- #252 owns `docs/planning/v0.6.0-responsive-visual-regression-suite.json` and `apps/web/e2e/responsive-visual-regression.spec.ts`: responsive route changes must preserve the required viewport/state matrix, runtime screenshot attachments, fixed header/bottom-nav clearance, and 320 CSS px no-horizontal-scroll guarantees without introducing committed image baselines.
- #256 owns the release documentation current-state artifacts: `docs/planning/CURRENT_STATE.json`, `docs/planning/snapshots/v0.5.0-current-state-archive.json`, `docs/planning/snapshots/v0.6.0-documentation-candidate-snapshot.json`, and `docs/planning/v0.6.0-documentation-screenshots.json`. Keep them aligned with the task-first UI, guide-map references, open release gates, and metadata-only screenshot policy.
