# Model storage

- Store model files behind a UI-independent backend interface; prefer OPFS when available, Cache Storage as a browser fallback, and in-memory storage only for tests or unsupported contexts.
- Keep service-worker app-shell caching separate from model-version storage; a service-worker update must not delete installed model files.
- Normalize storage locators as `<modelId>/<version>/<fileKey>` path segments and reject unsafe segments before writing.
- Never overwrite active model files in place; write temporary versions, verify downloaded and stored bytes, copy into the inactive target version, then update the active pointer last.
- For V3 browser-training manifests, ordinary model installs must store only ASR inference graph files by default. Optional training companion artifacts may be installed later only for the exact active model version, using temporary staging, checksum verification, artifact license gates, and a registry-side `trainingCompanion` record; do not load or download them during ordinary inference. Readiness UI may inspect and display aggregate companion state/file counts/bytes through the model-lifecycle worker, but must not fetch companion bytes or expose model storage internals.
- Treat the active model registry as the source of truth for the selected version; a failed download, checksum, storage verification, or license gate must leave the previous active pointer intact.
- Delete and list operations must be scoped by model ID and version so rollback/reinstall flows can preserve unrelated model versions.
- Deleting an active model through lifecycle UI must remove both the active registry record and that active version's files; it must not affect unrelated model versions.
- Do not store private audio, transcripts, enrollment recordings, or profile artifacts in the model-file backend.
- Portable `.speechmodel` inner bundles and encrypted envelopes are export artifacts, not model-cache entries. Generate them from explicit local export state, enforce hostile path/case-collision checks, reject default payload paths/media types that indicate raw audio, prepared feature shards, checkpoints, optimizer state, or training-job internals, and never persist passphrases or derived keys in OPFS/Cache/localStorage.
- Profile storage is separate from model storage: keep accepted-take WAV/metadata, export packages, browser-training private FP16 feature shards, private checkpoints, activation-review guarded active/previous profile pointers, and rollback state under `@speech/profile-manager` paths with safe segments, checksum verification, atomic writes, explicit import/export, and deterministic delete-all behavior. Synthetic browser-training UI demos may use browser-local storage for reload recovery plus redacted cross-tab/ASR-priority status, but private feature shards and checkpoints must not use the model-file backend. Stored browser-training checkpoints must validate resume-state checksums plus dataset, training-config, and sampler fingerprints before continuing a run; runtime pause/cancel/error/completion paths must dispose worker sessions and release mutable buffers after checkpoint/result creation. Recovery UI should summarize status/epoch/update time and never render checkpoint IDs, storage paths, checkpoint bytes, or adapter weights.
- Browser-training feature-shard manifests should be committed only after shard bytes are written and verified; summaries may expose aggregate counts/bytes/checksums and selected-vocabulary counts, while raw shard paths tied to prompt IDs and raw selected-vocabulary entry IDs stay profile-store local.
- Browser-training frame-label alignment sets live under the owning training-job feature set, bind the feature manifest checksum, preserve raw selected-vocabulary entry IDs only in private local label metadata, verify label-file and manifest checksums, stay outside default profile exports, and delete with the feature set/profile.
