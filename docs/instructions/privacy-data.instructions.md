# Privacy and data rules

- Audio and transcripts remain local unless the user explicitly exports them.
- v0.6 progressive-disclosure work must keep privacy consequences visible at the point the user records, trains, imports, exports, downloads diagnostics/support data, deletes local data, activates, or rolls back; hashes, schema versions, encryption parameters, and compatibility details may move behind details, but the privacy consequence and safe next action must not be hidden only in a tooltip, toast, or collapsed panel. Disclosure/Accordion/MenuButton/Tooltip/Dialog/Select/RadioGroup/feedback primitive examples and tests must reinforce that required privacy consent, destructive consequences, blockers, and recovery actions stay outside collapsed content, menus, tooltips, toasts, and advanced dialog content; dialogs may block briefly for confirmation/passphrase/rename decisions, notices may carry visible blockers, inline errors may carry field recovery, and long privacy or data-management workflows require dedicated screens. ADR 0012 copy-budget work may shorten privacy copy, but any exception or rewrite must preserve the local-only consequence and recovery/action wording required by `docs/planning/v0.6.0-terminology-copy-budgets.json`. ADR 0013 route work may preserve profile/vocabulary/job/query IDs only after syntax and domain-store validation; reject open redirects, unsafe external return targets, and query values that could expose private storage paths or untrusted HTML.
- `UiPreferencesV1` is presentation-only. Do not store personal audio, transcripts, vocabulary terms, profile/model IDs, job IDs, storage paths, passphrases, diagnostics, or portable-bundle data in the UI-preferences record; reset/delete UI preferences without deleting or exporting voice data.
- No telemetry, remote logging, analytics, or crash uploads are enabled by default.
- During active transcription, do not fetch remote services; model downloads and app updates must be explicit lifecycle events outside active dictation. Deployment CSP `connect-src` may allow the exact public origin needed for committed installable model files, but that exception is only for user-initiated model downloads and must not become telemetry, analytics, remote logging, or a broad third-party network allowance.
- Enrollment recordings, speaker embeddings, adapters, and exported profiles are sensitive voice-biometric-like personal data; follow `docs/instructions/voice-profile-threat-model.instructions.md` for threat-model reviews and release gates.
- Persist enrollment audio only after explicit user acceptance/save into private profile storage; never use localStorage/IndexedDB for raw voice data, and keep delete-all voice-data flows local and deterministic.
- v0.6 enrollment prompt screens may display current prompt text, progress, voice-condition label, aggregate accepted counts, and concise mapped quality feedback, but must not expose raw quality reason-code strings, low-confidence implementation wording, worker errors, raw utterance IDs, checksums, storage paths, audio bytes, or transcript/prompt metadata in default status. Low generic-recognizer confidence should be phrased as review/acceptance guidance, not rejection. Detailed SNR, clipping, VAD, alignment, pace, confidence, checksums, and stored-recording summaries belong behind `Recording details`, while privacy/recording consequences stay visible at the action point.
- Never place private audio/transcripts/profile artifacts in logs, screenshots, fixtures, support bundles, or CI artifacts.
- Base-vs-profile evaluation reports may be exported only as aggregate local metrics with privacy flags; omit prompt text, raw audio, raw embeddings, profile files, raw selected-vocabulary entry IDs, private vocabulary strings, and model weights.
- Profile export/import must be explicit user action, local-only, checksummed, and visibly sensitive because exported packages can include enrollment recordings, prompt text, microphone metadata, and derived profile files; deletion must clear stored raw and derived files plus active-profile pointers.
- v0.6 app shell/navigation may display only destination labels, app-menu destinations/actions, and aggregate local/offline/model-download/update status. Route migration may preserve profile/vocabulary/job/candidate IDs only after syntax and domain-store validation; redirect helpers must reject external return targets, traversal, unsafe schemes, storage paths, untrusted HTML, and unknown query keys. Do not place raw profile IDs, vocabulary IDs/terms, transcript text, storage paths, model hashes, model names, runtime versions/providers, storage totals, error stacks, or diagnostic payloads in persistent navigation, Local status summaries/popovers, screenshots, logs, or route-state tests beyond synthetic placeholder IDs needed to prove sanitization. The Dictate workspace may show user-edited transcript text in the editable transcript area and locally initiated copy/download flows only; toolbar/status/detail summaries must avoid model hashes, runtime provider strings, storage paths, transcript logs, and diagnostic payloads by default. The compact Vocabulary list may display user-entered set names and aggregate on/off/term counts, but import/export controls must remain explicit and warn that downloaded/pasted files can contain sensitive names or project terms. Vocabulary editor advanced disclosures may show local prompt previews and diagnostics for the active set, but default list/editor summaries should not expose tokenizer internals, raw entry IDs, or misleading static boundary messages.
- Dictate setup cards may show public model display name, exact version, aggregate download size, license, provenance, public notes, progress phase, and retry/install actions only. Keep checksums, storage paths, active registry records, worker error stacks, backend kind, and companion-pack internals out of the default setup UI and screenshots; microphone permission copy remains separate and appears only on the first recording attempt after setup is ready.
- Simplified microphone/offline/update states may show aggregate readiness, blockers, and recovery actions only. Do not render raw `DOMException` names, worklet messages, service-worker cache internals, model hashes, provider strings, catalog storage paths, or update stack traces in default UI or screenshots. `/settings/audio` may show input-test status, browser audio-processing choices, and aggregate calibration state by default; actual sample rates, track settings, noise/level readings, and AudioWorklet counters belong in `Advanced audio diagnostics`, and no microphone-check audio is persisted. `/settings/privacy` must keep the local-only statement, export/delete/diagnostics/docs controls, no-telemetry/no-account network result, and local-data consequences visible without relying on tooltips. `/settings/shortcuts` may document keyboard behavior by scope only; do not include raw profile/model/vocabulary IDs, transcript text, or hidden privacy/destructive instructions in shortcut copy. `/settings/diagnostics` support bundles and copied diagnostics must stay aggregate-only and exclude audio, transcript text, vocabulary terms, profile identifiers, model weights, storage paths, passphrases, and raw error stacks. `/about` may link to source, licenses, notices, and model provenance, but must not include private local-data summaries or marketing claims.
- Personal Models compact rows, row menus, readiness preflight, `/models/new` create-model drafts, `/models/:profileId/train` readiness screens, and detail summaries/accordions may display aggregate local profile/vocabulary/speech-model/capability/training-support/runtime-self-test status and missing-recording tasks only. User-facing model/vocabulary status, validation, import/export, and lifecycle copy should be selected by stable typed reason-code mappings in `apps/web/src/content/reasonCodes.ts`, not raw package/worker exception messages. The `/models/new` wizard may store only UI draft choices such as display name, speech-language target, mixed-language preference, and recording plan; never store raw audio, prompt text, transcript text, model weights, profile JSON, profile IDs, storage paths, or diagnostic data there. Do not render raw audio, transcript/prompt text, feature tensors, checkpoints, adapter weights, profile JSON, raw profile IDs, model IDs, hashes, storage paths, prompt IDs, vocabulary entry IDs, private vocabulary terms, or default WER/CER/gate/adapter/OPFS/WebGPU/WASM/base-model implementation wording in model-list rows, cards, detail summaries, accordion sections, readiness screens, preflight summaries, screenshots, logs, or reports. Readiness blockers and primary actions must stay visible; passed checks, support-file details, and privacy explanations may move behind disclosures. Row menus may expose low-frequency lifecycle actions but destructive/export/deactivate consequences need explicit confirmations or follow-up UI; hard blockers and incompatibilities stay visible above collapsed sections. The `/models/import` route must show only aggregate file/encryption/size/name/base/vocabulary-included/smoke status; do not expose raw bundle IDs, profile IDs, hashes, storage paths, private vocabulary terms, adapter values, archive file paths, passphrases, or raw importer errors in default copy. The `/models/:profileId/export` route must show aggregate name/speech-model/vocabulary/size/security status plus the visible privacy statement that recordings and training checkpoints are excluded; package/encrypt only in the worker/package layer, clear selected export bytes/passphrases after save or route changes where practical, keep manifest hashes/encryption parameters under Details, and keep legacy `.speechprofile` export explicitly separated because it may include sensitive enrollment data. The `/settings/storage` route may show aggregate byte totals/counts and reliable quota only; deletion confirmations must name removed and retained categories without raw profile/model IDs, hashes, storage paths, private vocabulary terms, worker errors, or raw filenames.
- Multi-profile import/dedupe/rename/delete UI must keep raw source/target profile IDs inside worker/package calls. User-visible import results should describe the operation and collision handling only; per-card actions may use profile IDs internally but must render display names and aggregate status, not raw storage paths or identifiers.
- Local trainer dataset loaders may read exported enrollment audio and prompt text only from explicit user-provided profile packages, must not print or fixture private content, and must keep train/validation/test leakage checks local to the user machine.
- Local trainer adapter metadata may include aggregate split counts, hashes, selected-vocabulary ID hashes/counts, losses, adapter checksums, and software/config provenance, but must not include raw audio, prompt/reference transcript text, full profile JSON, base weights, raw selected-vocabulary entry IDs, or private vocabulary strings.
- Adapter evaluation and activation-gate reports may include aggregate WER/CER/custom-term/false-insertion/RTF metrics, selected-vocabulary counts, adapter size/checksum, and pass/fail checks only; omit raw audio, transcripts, case IDs, raw selected-vocabulary entry IDs, private vocabulary strings, full profile JSON, and model weights.
- Personal/anchor end-to-end evaluation reports may compare `generic`, `p1`, and `candidate` configurations using aggregate metrics, slice counts, adapter size/checksum, and pass/fail gates only. They must not expose raw case IDs, prompt/reference text, audio, profile JSON, feature tensors, checkpoints, adapter weight values, or raw selected-vocabulary entry IDs/terms.
- Personal-model activation decisions and comparison/result UI may display aggregate quality-check status, hard/advisory classification, selected-vocabulary counts, aggregate model size, processing overhead, and rollback/fallback availability only. Candidate result screens must lead with outcome/next action, group exact metrics under disclosed Results sections, and keep activation disabled on hard-check failure without revealing raw profile IDs, prompt IDs, case IDs, transcript text, feature tensors, checkpoints, adapter weights, vocabulary terms, raw vocabulary entry IDs, hashes, or storage paths. Advanced override copy must not reveal raw profile IDs, prompt IDs, case IDs, transcript text, feature tensors, checkpoints, adapter weights, vocabulary terms, or raw vocabulary entry IDs.
- Personal adapter packages are explicit sensitive exports; they may include adapter weights and a profile manifest, but must omit raw audio, transcripts, case IDs, full enrollment profile JSON, and base model weights while preserving checksum/base-model compatibility metadata.
- `SpeechProfileManifestV2` migration and `browser-top-adapter` adaptation are metadata-only; the manifest must never carry raw audio, transcripts, frozen-feature values, checkpoints, or optimizer state, and the copy-on-write V1->V2 migration must not rewrite or convert existing CLI residual adapters. Migration/recovery summaries may expose only aggregate schema/adaptation/base-model/version status and must not include raw profile IDs, storage paths, adapter file paths, adapter bytes, private vocabulary, prompt text, or full profile JSON. Browser adapter weight file refs are voice-derived sensitive data and follow the same export/import/deletion boundaries as other adapter artifacts.
- Portable `.speechmodel` bundles must exclude raw enrollment audio, prepared features, optimizer/checkpoint state, and the shared base model; treat adapter weights and speaker embeddings as voice-derived sensitive data, encrypt exports by default with AES-256-GCM + PBKDF2-HMAC-SHA-256 (>=600,000 iterations), never store the passphrase, clear passphrase byte copies after derive/encrypt/decrypt attempts, support cancellation checks before/after Web Crypto operations, and treat every imported file as hostile (validate magic/size before allocation, enforce file-count/path/compression/tensor limits, reject absolute/parent/backslash paths, reject base-model/operator/external-data payloads and raw-data privacy flags, enforce exact active base-model identity, run local worker-owned smoke vectors with timeout cleanup, and stage under temporary profile storage before writing the committed import record last). Import-security corpus and fuzz-style fixtures must be synthetic and must not contain raw audio, transcript text, feature tensors, checkpoints, adapter weight values, private vocabulary terms, real profile IDs, or real passphrases. Deterministic inner bundles may include adapter weights, optional speaker/vocabulary files, aggregate evaluation, notices, checksums, synthetic test vectors, and for CLI adapters the original residual-adapter profile manifest as metadata only, when manifest refs and archive index checksums match exactly and no path/media type indicates raw audio, prepared features, checkpoints, optimizer state, base-model graphs, operator files, external data, or training-job internals. Local trainer `.speechmodel` output is allowed only through an explicit unencrypted-export opt-in and must not include raw prompt text, audio, private vocabulary terms, feature tensors, or base weights. Browser `.speechmodel` exports must default to encryption, reject non-exportable enrollment-only profiles, and include only checksummed residual/browser-top adapter manifests/artifacts plus aggregate metadata needed for compatibility and smoke validation.
- Browser-training spikes, backend selectors, and fault-injection E2E suites may inspect package metadata, export maps, public type declarations, aggregate capability flags, metadata-only worker proof status, and synthetic/local UI state only; they must not persist or upload raw audio, transcripts, frozen features, profile contents, checkpoints, or adapter bytes, and injected errors must not echo private profile IDs or artifact paths.
- Bilingual 30-speaker cohort release evidence must be user-approved and aggregate-only. Do not commit raw speaker identifiers, profile IDs, prompt/case IDs, transcript text, audio, feature tensors, checkpoints, adapter weights, or private vocabulary; if evidence is missing, commit only an insufficient-evidence gate summary.
- Personal-model reference benchmark evidence must be aggregate-only and local. Do not commit raw audio, transcript text, feature tensors, checkpoints, adapter weights, profile JSON, private vocabulary, prompt/case/profile IDs, storage paths, or network payloads; if declared reference-hardware evidence is missing, commit/export only an insufficient-evidence benchmark gate summary.
- Dictate performance parity reports must also be aggregate-only and local. Do not include raw transcript text, microphone audio, profile/vocabulary identifiers, storage paths, URLs, screenshots, provider internals, or diagnostic payloads; CI/fake-microphone measurements may be exported only as insufficient-evidence instrumentation when real-ASR or v0.5 reference baselines are missing.
- Release privacy/security/licensing reviews should be recorded as ADRs when they affect release claims. They may summarize aggregate review outcomes and blockers, but must not include raw voice/profile artifacts, storage paths, private identifiers, real passphrases, or unpublished license-sensitive materials.
- Model manifest V3 browser-training contracts are public compatibility metadata only. They may name tensors, file keys, licenses, provenance, algorithm IDs, and limits, but must not contain user recordings, transcripts, frozen-feature values, checkpoints, adapter weights, private vocabulary terms, or profile-derived artifacts.
- `@speech/browser-training` prototypes may emit aggregate loss/parameter/checksum metadata and non-sensitive synthetic fixtures through the `BrowserTrainingBackend` interface; private frozen features or profile-derived datasets must stay worker-local/OPFS-local and must not appear in logs, screenshots, fixtures, telemetry, or exports unless explicitly packaged by the user. Sampler, optimizer, validation, early-stopping, and safety diagnostics may include counts, losses, bucket totals, learning-rate metadata, and clip-event counts only; do not expose raw feature tensors, transcript text, prompt IDs, checkpoint bytes, adapter weights, or private vocabulary terms in diagnostics.
- Training-readiness coverage reports may include aggregate counts, durations, pass/fail requirements, and redacted prompt/vocabulary labels only; omit raw prompt IDs, transcript text, vocabulary terms, audio, feature tensors, checkpoints, adapter weights, and raw profile JSON.
- Frozen browser-training job revisions are local profile-store records, not default exports. They may persist active vocabulary terms locally to make the training input immutable, but verification reports must stay aggregate/redacted, profile exports must not include training-job revisions by default, and no revision content may be logged, screenshotted, uploaded, or exported without an explicit user action.
- Browser-training prompt split reports may include aggregate split counts, durations, language/voice-condition counts, seeds, and redacted prompt labels only; raw prompt IDs, prompt text, audio refs, vocabulary terms, frozen features, checkpoints, and adapter weights must remain in local worker/profile-owned storage unless explicitly exported by the user.
- Browser-training FP16 feature-shard summaries may include aggregate utterance/frame/shard counts, byte totals, split totals, and checksums only. Raw feature tensors, prompt IDs, utterance IDs, shard paths, audio bytes, transcript text, vocabulary terms, checkpoints, and adapter weights must stay in local profile/feature-preparation worker storage and be deleted with the owning voice profile unless the user explicitly exports a future sensitive training bundle.
- Browser-training frame-label summaries may include aggregate aligned/excluded utterance and frame counts plus selected-vocabulary counts only. Token IDs, frame weights, masks, confidence values, prompt IDs, utterance IDs, logits, raw selected-vocabulary entry IDs, audio, transcript text, vocabulary terms, feature tensors, checkpoints, and adapter weights remain private profile/feature-preparation worker data; low-confidence exclusions must mask training frames without deleting the accepted recording.
- Browser-training checkpoints can contain adapter weights, raw local resume parameters, optimizer moments, sampler/config fingerprints, validation state, and checksums derived from frozen features. The public UI prototype may persist synthetic checkpoints in browser-local storage for reload recovery, but private/profile checkpoints must remain in dedicated worker/profile-owned storage and must not be logged, screenshotted, uploaded, exported by default, or treated as activated profiles. Training progress and recovery UI, including accessible names, live regions, confirmations, and visual text equivalents, may show only user-facing stage names (Preparing, Training, Checking, Ready), aggregate progress, sanitized resource guidance, and local recovery summaries by default; epoch/loss/LR/backend/thread/memory/checkpoint metrics belong in `Training details`. Omit checkpoint IDs, checkpoint bytes, adapter weights, feature tensors, raw profile IDs, prompt IDs, and private vocabulary data from both default progress and details.
- Browser-training cross-tab coordination may broadcast local lock lifecycle events only with redacted scope fingerprints and aggregate messages. Do not broadcast raw profile IDs, dataset IDs, prompt IDs, transcripts, feature tensors, checkpoints, adapter weights, private vocabulary IDs/terms, or profile JSON.
- ASR-priority broadcasts may expose only local active/idle state, coarse reason codes, timestamps, and aggregate messages. They must not include utterance IDs, raw audio, transcript text, profile IDs, dataset IDs, feature tensors, checkpoints, adapter weights, private vocabulary IDs/terms, or profile JSON.
- Browser-vs-Python adapter comparison exports may include aggregate metric deltas, activation-gate flags, adapter checksums/sizes, and base-model compatibility only; omit transcript text, case IDs, frozen-feature values, profile JSON, adapter weights, raw audio, and private vocabulary strings.
- Local trainer Docker images must not bake exported profiles, raw recordings, transcripts, adapter outputs, aggregate evaluation inputs, or model weights into image layers; runtime examples should prefer `--network none`, narrow bind mounts, and host UID/GID mapping.
- Custom-vocabulary prompt generation uses user-entered terms, aliases, and categories as local sensitive data; generated prompts must stay local by default, be shown for user review before recording, and not be committed as release fixtures or uploaded.
- Code license, model license, dataset license, and fixture license are separate release gates.
- Catalog research leads with unclear or `other` license metadata must stay blocked/non-installable and must not be downloaded by the PWA until redistribution and runtime compatibility are documented.
- Data-governance docs must state source, license/consent, speaker/source split policy, and redistribution rights before adding training audio, pseudo-labels, corpora, or derived weights.
- A few minutes of enrollment adapts a pretrained model; never describe it as training a new ASR base model from scratch.

- The v0.6 component gallery and usage documentation must use synthetic examples only. It must not render or store real profile IDs, prompt IDs, transcripts, personal audio, private vocabulary terms, feature tensors, checkpoints, adapter weights, passphrases, or storage paths; required privacy and destructive/recovery consequences remain visible in examples rather than hidden only in tooltips, menus, toasts, or collapsed panels.
