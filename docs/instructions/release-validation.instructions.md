# Release validation rules

- Keep CI release-validation checks deterministic and bounded. They may stress rapid controls and synthetic benchmarks, but multi-hour soak and hard latency gates belong on declared reference hardware.
- Accessibility validation must fail for unnamed interactive controls, unlabeled main sections, duplicate IDs, or broken keyboard focus through primary actions.
- Active transcription validation must assert no fetch, XHR, or websocket requests while push-to-talk is held. Scope network/console-error collection to the active transcription window so favicon or app-shell preload noise is not mistaken for live-audio telemetry. Do not infer local-processing privacy from copy alone.
- Short CI stress cycles should catch repeated start/stop regressions, page errors, console errors, and continuously increasing queue/chunk counters; they are not a substitute for 30-minute or one-hour manual/reference runs.
- Performance validation must verify benchmark exports include first partial latency, stable token latency, finalization latency, encoder/decoder chunk timing, RTF, queue depth, overrun count, privacy flags, and interpretation warnings.
- Dictate performance-parity smoke may measure initial JS/CSS resource bytes, interaction readiness, route transition, long tasks, CLS, recording UI response, and observed latency fields in CI, but release parity remains insufficient until v0.5 reference baselines and real-ASR reference-hardware latency evidence are available.
- Validation fixtures must remain synthetic or generated by browser fake devices; do not add private audio, transcripts, screenshots with personal data, or production model weights to CI artifacts.
