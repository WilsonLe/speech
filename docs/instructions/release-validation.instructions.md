# Release validation rules

- Keep CI release-validation checks deterministic and bounded. They may stress rapid controls and synthetic benchmarks, but multi-hour soak and hard latency gates belong on declared reference hardware.
- Accessibility validation must fail for unnamed interactive controls, unlabeled main sections, duplicate IDs, or broken keyboard focus through primary actions.
- Active transcription validation must assert no fetch, XHR, or websocket requests while push-to-talk is held. Scope network/console-error collection to the active transcription window so favicon or app-shell preload noise is not mistaken for live-audio telemetry. Do not infer local-processing privacy from copy alone.
- Short CI stress cycles should catch repeated start/stop regressions, page errors, console errors, and continuously increasing queue/chunk counters; they are not a substitute for 30-minute or one-hour manual/reference runs.
- Performance validation must verify benchmark exports include first partial latency, stable token latency, finalization latency, encoder/decoder chunk timing, RTF, queue depth, overrun count, privacy flags, and interpretation warnings.
- Dictate performance-parity smoke may measure initial JS/CSS resource bytes, interaction readiness, route transition, long tasks, CLS, recording UI response, and observed latency fields in CI, but release parity remains insufficient until v0.5 reference baselines and real-ASR reference-hardware latency evidence are available.
- Validation fixtures must remain synthetic or generated by browser fake devices; do not add private audio, transcripts, screenshots with personal data, or production model weights to CI artifacts.
- The opt-in production dictation smoke (`SPEECH_PRODUCTION_DICTATION_E2E=1`) must hit the deployed app by default, download the real model without route/fulfill/seed shortcuts, generate fake-microphone audio locally, redact signed model URLs in artifacts, and distinguish model-install/download/CSP failures from real ASR/transcription mismatches. Normal CI/dev runs must skip it.
- The #253 manual accessibility matrix lives in `docs/planning/v0.6.0-manual-accessibility-matrix.json` and is enforced by `tests/static/test_v06_manual_accessibility_matrix.py` plus `apps/web/e2e/manual-accessibility-matrix.spec.ts`. Keep axe critical/serious route scans, keyboard/focus-obscured checks, forced-colours, 200% text-scaling, live-region restraint, and no-hover-only primary-action checks current; document Windows/macOS AT rows as explicit accepted limitations when NVDA/Edge/VoiceOver/Safari cannot run in the Linux agent.
- The #254 prototype usability artifact lives in `docs/research/v0.6-prototype-usability-study.json` with summary `docs/research/v0.6-prototype-findings.md`. If consented participant sessions are unavailable, record the blocker and keep the usability gate open; do not claim participant findings or release readiness from semantic/visual/accessibility substitute evidence alone.
- The #255 release usability artifact lives in `docs/research/v0.6-release-usability-study.json` with summary `docs/research/v0.6-release-usability.md`. If final moderated participant evidence is unavailable, record the insufficient-evidence blocker, keep issue #255 and the release usability gate open, and never mark 90% completion, comprehension, hover-only, or high-severity-finding gates as passed without consented participant evidence or an explicit human release decision.
- The #257 update/offline/rollback artifact lives in `docs/planning/v0.6.0-update-offline-rollback-verification.json`. Keep it limited to local same-origin two-version PWA evidence, explicit service-worker activation/rollback checks, additive UI-preference compatibility, and residual limitations for issue #258 hosted release/deployment plus the open #255 usability gate.
